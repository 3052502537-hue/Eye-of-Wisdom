## 1. 设计概念

### 1.1 项目定位
"智慧之眼"是一款基于**双 ESP32-S3** 架构的互动式导盲头环，旨在为视障人士提供**全方位环境感知**与**智能出行辅助**能力。项目定位为大学生创新大赛参赛项目，重点验证多传感器融合（激光测距 + 毫米波雷达 + AI视觉）导盲方案的可行性，实现盲道检测、障碍物避障、交通灯识别等核心功能。

### 1.2 核心设计理念
- **安全优先**: 避障预警是第一优先级，响应速度优先于识别精度
- **多传感器融合**: 激光测距提供精确距离、毫米波雷达检测运动目标、AI视觉理解场景语义
- **端侧AI推理**: 视觉识别在手机端本地运行（TensorFlow Lite），隐私安全且支持离线使用
- **双板异构架构**: 主控板负责传感器与决策，摄像头板专注图像处理，SPI高速通信
- **可穿戴形态**: 头环式设计，解放双手，自然交互
- **手机联动**: 通过WiFi与手机APP互联，利用手机算力和TTS能力

### 1.3 目标功能（按优先级排序）

| 优先级 | 功能模块 | 功能描述 | 实现方式 |
|--------|----------|----------|----------|
| P0 | 激光测距避障 | 前方精确距离测量与预警 | SDM10 激光测距（22m量程）+ 蜂鸣器 |
| P0 | 毫米波雷达避障 | 前后方运动目标检测与接近预警 | Rd-03D 毫米波雷达（2个：前+后） |
| P0 | 盲道识别 | 识别地面盲道区域，偏离提醒 | OV2640摄像头 + TFLite AI识别 |
| P1 | 交通灯识别 | 识别路口交通信号灯状态 | 摄像头 + TFLite AI识别 |
| P1 | 障碍物检测 | 检测前方大型障碍物（车辆、行人等） | 摄像头 + TFLite AI识别 |
| P1 | 路口识别 | 识别十字路口、丁字路口 | 摄像头 + TFLite AI识别 |
| P2 | 语音导航 | 语音播报环境信息和导航指引 | 手机APP TTS |
| P2 | 风险分级播报 | 安全/注意/危险三级预警 | 蜂鸣器频率 + 语音优先级 |
| P2 | 远程监控 | 亲友实时查看用户位置和状态 | 手机APP |

> **说明**: P0 为必须实现的核心功能，P1 为重要功能，P2 为扩展功能。比赛项目优先完成 P0 + P1。

### 2.2 关键设计决策

#### 决策1：为什么采用双板架构？
| 方案 | 单板 ESP32-S3 | 双板 ESP32-S3 |
|------|-------------|--------------|
| 算力分配 | 摄像头+传感器+通信争抢同一CPU | 主控板专注传感器和决策，摄像头板专注图像 |
| 内存压力 | PSRAM被图像独占，其他任务内存紧张 | 两板独立内存，互不干扰 |
| SPI通信 | 无 | 80Mbps高速图像传输 |
| 开发复杂度 | 低 | 中（需联调） |
| 扩展性 | 有限 | 好（可独立升级） |

**结论**: 双板架构分工明确，避免算力和内存瓶颈，图像传输效率高，选择双板方案。

#### 决策2：激光测距 + 毫米波雷达的互补优势
| 传感器 | 优势 | 劣势 | 用途 |
|--------|------|------|------|
| SDM10激光 | 精度高(±2cm)、量程远(22m)、方向性强 | 仅单点、无法检测运动 | 前方精确距离测量 |
| Rd-03D毫米波 | 检测运动目标、速度感知、扇形覆盖(±60°) | 精度较低(±10cm)、量程近(8m) | 前后方运动目标检测 |

**结论**: 激光测精确距离 + 雷达测运动状态，二者互补，提供完整的环境感知能力。

#### 决策3：AI推理放在手机端
| 方案 | ESP32-S3端侧推理 | 手机端TFLite推理 |
|------|-----------------|-----------------|
| 算力 | 有限，只能跑微型模型 | 强大，可跑中型模型 |
| 识别精度 | 较低 | 较高 |
| 隐私安全 | 本地运行，安全 | 本地运行，安全 |
| 模型更新 | 需重新烧录固件 | APP更新即可 |
| 开发难度 | 中（需适配TFLite Micro） | 低（标准TFLite） |

**结论**: 利用手机强大算力运行TFLite模型，识别精度更高，开发更简单。

#### 决策4：WiFi AP模式通信
| 模式 | Station模式 | AP模式 |
|------|------------|--------|
| 连接方式 | ESP32连路由器 | 手机直接连ESP32热点 |
| 依赖网络 | 需要路由器 | 不需要 |
| 适用场景 | 室内/有WiFi环境 | 室外/无网络环境 |
| 延迟 | 高（经过路由器） | 低（直接连接） |

**结论**: AP模式无需依赖外部网络，响应延迟低，更适合户外导盲场景。

### 2.3 预警策略设计

#### 距离分级预警（激光主导）
| 距离范围 | 危险等级 | 反馈方式 |
|----------|----------|----------|
| > 5米 | 安全 | 无反馈 |
| 3 ~ 5米 | 注意 | 慢速蜂鸣(1Hz) |
| 1 ~ 3米 | 警告 | 中速蜂鸣(2Hz) |
| < 1米 | 危险 | 急促蜂鸣(4Hz) + 语音播报 |

#### 运动目标预警（雷达主导）
| 条件 | 预警类型 | 反馈方式 |
|------|----------|----------|
| 后方目标快速接近(>10km/h) | 紧急预警 | 蜂鸣器连续响 + 语音"后方来车" |
| 前方目标移动 | 注意预警 | 蜂鸣器提示 |
| 静止目标 | 距离预警 | 根据距离分级 |

#### APP三种工作模式
| 模式 | 功能 | 功耗 | 适用场景 |
|------|------|------|----------|
| 传感器模式 | 仅激光+雷达数据，无AI识别 | 低 | 熟悉环境、省电 |
| 自动模式 | 全功能：传感器+AI识别+语音导航 | 中 | 陌生环境、完整导航 |
| 风险播报模式 | 仅播报安全/注意/危险三级 | 低 | 安静环境、信息精简 |

### 3.4 SPI板间通信引脚 (确定版)

| 信号 | 主控板 (SPI Master) | 摄像头板 (SPI Slave) | 方向 |
|------|--------------------|---------------------|------|
| MOSI | GPIO11 | GPIO41 | 主控→摄像头 |
| MISO | GPIO13 | GPIO42 | 摄像头→主控 |
| SCK | GPIO12 | GPIO40 | 主控→摄像头 |

| -1 | RESET | 复位(未接) |

**SPI Slave接口 (与主控板交叉连接):**

| 引脚 | 功能 | 连接至主控 |
|------|------|-----------|
| GPIO41 | SPI_MOSI | 主控GPIO11 |
| GPIO42 | SPI_MISO | 主控GPIO13 |
| GPIO40 | SPI_SCLK | 主控GPIO12 |
| GPIO39 | SPI_CS | 主控GPIO10 |
| GPIO38 | DATA_READY | 主控GPIO9 |

**电源与状态:**

| 引脚 | 功能 |
|------|------|
| 3.3V | 摄像头+ESP32供电 |
| GND | 必须与主控共地 |
| GPIO48 | 状态LED(可选) |

> **引脚规划原则**: DVP数据线使用GPIO8-18连续区域；SPI使用GPIO38-42高编号区域；避开OPI PSRAM(GPIO26-37)、启动引脚(0,3,45,46)、USB(19,20)。

### 4.4 电源设计注意事项

1. **18650锂电池**容量2600mAh，传感器模式续航>12小时，全功能模式4-6小时
2. **5V输出**供SDM10激光和Rd-03D雷达（雷达峰值电流≤200mA）
3. **3.3V输出**供ESP32主控板和OV2640摄像头
4. 各模块电源入口加 **100nF去耦电容 + 10μF滤波电容**
5. **共地必须良好**，两块板子GND必须连在一起，否则SPI通信异常
6. 电源管理芯片需支持至少 **1A** 输出电流

### 6.2 通信协议定义

#### 6.2.1 TCP JSON数据格式（传感器数据，确定版）

```json
{
  "type": "sensor",
  "ts": 1620000000,
  "laser_front": 3.25,
  "radar_front": {"dist": 1.5, "speed": 0.8, "angle": 15.0},
  "radar_back":  {"dist": 2.0, "speed": -0.5, "angle": -30.0},
  "battery": 85,
  "mode": 2,
  "level": 0,
  "img": 1
}
```

> 字段名与Android APP `Protocol.SensorData` 一致。`laser_front`为数值型(m), `radar_front/back`为对象(取最近目标)。`level`: 0=安全/1=注意/2=危险。`img`: 0/1表示是否有新图像帧。

#### 6.2.2 UDP二进制格式（图像数据）

| 字段 | 长度 | 说明 |
|------|------|------|
| 帧头 | 2B | 0xAA55 |
| 帧序号 | 4B | 递增序列号 |
| 总帧数 | 4B | 该JPEG的总片数 |
| 当前帧 | 4B | 当前分片序号 |
| 数据长度 | 4B | 本次数据长度 |
| 数据 | 可变 | JPEG数据 |
| CRC8 | 1B | 校验和 |
| 帧尾 | 2B | 0x55AA |

#### 6.2.3 SPI帧格式（板间通信，确定版）

| 字段 | 长度 | 说明 |
|------|------|------|
| 帧头 | 2B | 0xAA55 |
| 命令 | 1B | START(0x01)/DATA(0x02)/END(0x03)/HEARTBEAT(0x30) |
| 长度 | 2B LE | 数据长度(小端) |
| 数据 | 可变 | 图像数据或配置参数 |
| CRC8 | 1B | 校验和(多项式0x07) |
| 帧尾 | 2B | 0x55AA |

> 帧开销=8字节。图像传输3步协议: START(元数据)→DATA(分块+blockIndex/blockSize)→END(totalBlocks)。

### 6.3 关键算法逻辑

#### 6.3.1 激光测距算法（核心）
1. **数据获取**: 通过UART1读取SDM10激光传感器数据
2. **协议解析**: 解析激光传感器输出的距离值（单位：米）
3. **距离分级**: 根据距离值触发不同等级预警（安全/注意/警告/危险）
4. **滤波处理**: 简单滑动平均滤波，减少抖动

#### 6.3.2 毫米波雷达算法（核心）
1. **数据获取**: 通过UART0/2读取Rd-03D雷达数据
2. **协议解析**: 解析帧头0xAA 0x55，提取目标列表（距离、角度、速度）
3. **目标过滤**: 过滤无效目标（距离<0.5m或>8m）
4. **危险判定**: 前方目标按距离分级，后方目标关注接近速度
5. **时间滤波**: 连续2~3帧确认同一目标才触发预警

#### 6.3.3 TFLite AI视觉识别（手机端）
1. **模型架构**: YOLOv8轻量化模型，转换为TFLite格式
2. **识别类别**: 盲道、交通灯、障碍物、路口、其他（共5类）
3. **推理优化**: INT8量化、GPU加速、多线程推理
4. **输出格式**: 类别ID、置信度、边界框坐标
5. **推理速度**: 中端手机10-15fps

#### 6.3.4 多传感器融合决策（手机端）
1. **数据对齐**: 基于时间戳对齐传感器数据和图像识别结果
2. **优先级仲裁**: 危险预警 > 视觉识别 > 距离提示
3. **风险分级**: 根据融合结果输出安全/注意/危险三级
4. **策略扩展**: 采用策略模式设计，支持算法替换

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
